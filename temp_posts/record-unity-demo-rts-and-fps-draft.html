Flow Field Path Finding 大致介绍

Flow Field（流场）是一种基于向量场的寻路方案，特别适合大规模单位的移动场景，比如 RTS 游戏中的几百上千个单位同时移动。 它的核心思想是：不计算每个单位的完整路径，而是预计算一个”流向场”，每个格子存储一个移动方向，单位只需要根据当前所在位置的向量方向移动即可。 这就像水流一样——你不需要计算每条水流的完整路径，只需要知道每个点的水流方向，水自然会流向低处。

为什么 RTS 需要 Flow Field？

传统的 A* 寻路在少量单位时没问题，但一旦单位数量变得庞大，如果每个单位都独立跑 A*，会带来巨大的计算开销（虽然也有只计算一次 A*，多单位跟随的方法）。 而 Flow Field 的特点是：

  • 一次计算，全员使用 —— 计算一次流向场，所有单位共享

实现 Flow Field 的基本步骤

  1. 将地图划分成网格（Grid）
  2. 根据地图地形生成 Cost Map
  3. 当玩家确定此次目的地以后生成 Heat Map
  4. 根据 Heat Map 生成 Flow Field

这只是博主的实现步骤，仅供参考哦。

数据结构定义

首先我们定义两个关键的 struct：DirectionCell（方向格子）和 ObstacleCell（障碍格子）。

DirectionCell 是整个系统的核心，每个格子存储了：

  • cost：移动代价。默认是 1，崎岖地形会变成 2，不可通行地带设为 float.PositiveInfinity
  • heat：热值。这个格子和目标点的”距离”（不是实际几何距离，而是经过障碍调整后的”热量”扩散）
  • direction：最终单位应该往哪里走的单位向量

FlowField 类详解

FlowField 是主要的类，负责所有网格的计算和生成。我们一步步来看。

构造与初始化

这里用了 NativeArray<T> 和 NativeParallelMultiHashMap<int, int>，这是在 Unity 的 Burst + Jobs System 中常用的类型，能够实现高性能并行计算。Allocator.Persistent 表示这些内存不会自动释放，需要我们手动 Dispose()。

  • directionGrid：存储所有方向格子
  • obstacleGrid：存储障碍格子（障碍物只需要位置）
  • cellToUnit：映射哪个格子里有哪些单位（用于单位快速查询）
  • cellToObstacle：映射哪个格子里有哪些障碍物

网格生成（GenerateGridBurst）

首先需要生成最初的网格数据，包括方向格子和障碍格子的基础信息（worldPos 等）：

这里使用了 Burst job 并行填充每个格子的坐标和默认值。

费用场生成（GenerateCostField）

这是最繁琐的一步。我们需要遍历每个方向格子，检测它是否被障碍物覆盖，从而设置 cost。

关键策略：sub-cell 九宫格细分

每个 DG 格子再细分为 3×3 的 sub-cell，通过 Physics.OverlapBox 检测整体碰撞，再对 sub-cell 做射线检测，从而更精确地判断格子是否”大部分”被障碍覆盖。

  • Impassible：若 ≥4 个 sub-cell 被 impassible 层阻挡 → 整格 cost = ∞
  • Rough：若 ≥4 个 sub-cell 被 rough 层阻挡 → 整格 cost = 2

优势：避免将边界或部分覆盖的障碍误判为全格障碍，提供更平滑的成本场。

Heat Map 生成（GenerateHeatMapBurst）

Heat Map 是流场寻路的核心步骤。它模拟热量从目标点向四周扩散的过程，每个格子获得一个 heat 值，heat 越小表示离目标越近。

算法：类似 Dijkstra 的带权重 BFS，使用 openList + closeList。

流程：

  1. 目标格子 heat = 0，入队
  2. 循环出队：对当前格子的 8 个邻居（含对角）尝试松弛
    newHeat = current.heat + neighbor.cost
  3. 若新值更小，更新 neighbor.heat 并入队
  4. 直到 openList 为空或找到可达终点
  5. 原地修改 directionGrid 中的 heat 字段

关键点：使用 NativeQueue 和 NativeArray<byte> 标记列表，全部在 Burst Job 中并行执行。

Flow Field 生成（GenerateFlowFieldBurst）

有了 heat 值之后，计算每个格子的最终移动方向：取周围 8 个邻居中 heat 最小的方向。

  • FlowFieldJob 并行计算每个格子的方向向量
  • 不可通行格子的方向保持 (∞, ∞)；目标格子方向为 (0,0)（用 Flag 材质显示）

障碍物映射（GenerateObstacleMap）

将动态障碍物（单位、移动物体）记录到 cellToObstacle，以便流场计算时避开。

这里维护了一个 colliderBuffer 缓存，避免每次调用 GetComponent<ObstacleAgent>().id 的开销。

GridController：控制器与交互

GridController 是 MonoBehaviour 脚本，负责把整个系统串联起来，并处理玩家输入。

初始化流程

所有初始化在主线程同步执行，使用 Burst Job 加速但 .Complete() 等待完成。

玩家点击处理

当玩家点击地面时：

  1. 将鼠标位置转换为世界坐标
  2. 转换为方向网格索引
  3. 生成 Heat Map（针对该目标点）
  4. 生成 Flow Field（所有单位共享的方向场）
  5. 发布 MoveToEvent 事件，通知各单位移动

两个生成步骤使用 Burst Job，总耗时通常 < 10 ms。

调试可视化

OnDrawGizmos 中用各类材质渲染箭头，显示每个格子的方向。这对调试非常有帮助！

障碍物系统详解

障碍物系统是流场寻路中处理地形阻塞的关键组件。主要有三个部分：ObstacleAgent（附加在障碍物上的 MonoBehaviour）、ObstacleData（数据组件）以及 ObstacleRegister（注册与管理映射）。

几何形状定义

ObstacleData 支持两种形状：

  • Circle（圆形）：中心点 + 半径，表示圆柱状障碍
  • Rectangle（矩形）：中心点 + 尺寸 + 朝向向量，表示轴对齐或旋转的矩形

流场不能直接用几何形状判断格子是否被占据，而是将每个障碍物在其覆盖的障碍网格单元中注册，然后 cellToObstacle 告诉流场”这些格子里有障碍”。

注册与注销

ObstacleAgent 在 Awake 中构建 ObstacleData，调用 ObstacleRegister.instance.Register(obstacle) 获取唯一 ID。OnDisable 中自动注销。还订阅 ObstacleRemoveEvent 用于 ID 变更同步，确保状态一致。

ObstacleRegister 维护一个从障碍网格索引到障碍ID列表的映射。初始化时 FlowField.GenerateObstacleMap 扫描所有静态障碍；动态障碍通过 Register/Unregister 实时更新。如果障碍移动，ObstacleRegister 会先从旧格子移除，再添加到新格子，保持映射最新。

与流场的协作

障碍映射并非直接用于计算流向，而是作为”成本加成”：在 GenerateCostField 阶段，系统检查每个方向格子是否被障碍物覆盖（通过 cellToObstacle 判断），然后根据覆盖的子细胞数提高该格子的 cost。不可通行区域标记为 float.PositiveInfinity，崎岖区域 cost 设为 2。这样 Heat Map 的扩散会自然绕过高成本区域，产生合理的流向。

单位系统与流场协同

单位系统是流场实现中”使用”这一端的核心。同样由 ECS+DOTS 构建：UnitAgent（MonoBehaviour 桥接层）、UnitAgentData（纯数据组件）以及 UnitBus（事件总线）。单位通过订阅 MoveToEvent 来响应玩家的移动指令。

当玩家在地图上点击时，GridController 会执行以下流程：

  1. 将点击的世界坐标转换为方向网格索引
  2. 调用 GenerateHeatMapBurst 计算从目标点出发的热力图
  3. 调用 GenerateFlowFieldBurst 生成每个格子的流向向量
  4. 发布 MoveToEvent，事件中携带目标位置

所有被选中的单位都会收到该事件。UnitAgent 在回调中只需要记住这个事件——实际上流场已经是全局最新状态，单位只需在每帧 Update 中查询自己当前所在格子的方向，然后沿着该方向前进。

核心更新逻辑（简化）：

这种设计的优势在于单位本身不需要进行任何寻路计算，它们仅仅是”乘客”，沿着流场这个”river”漂流。极端情况下，即使有成千上万个单位，它们的移动成本也只是每帧一次网格查询和一次向量移动，开销极低。

当然，光有全局流场还不够——单位之间可能会碰撞。常见的解决方案是在流场之上叠加一个局部避障层，例如使用 RVO（Reciprocal Velocity Obstacle） 或者简单的 分离力。在 DOTS 中，这些局部行为也可以用 Burst Job 并行处理，每个单位在读取流场方向后，再根据邻近单位的位置微调方向。我们将这些细节留到下一篇博客吧！

性能瓶颈与优化思路

虽然流场在大规模单位场景中表现卓越，但在实际项目中仍可能遇到性能瓶颈。以下是几个常见问题及优化方向：

  • 网格分辨率与内存开销：网格越细，路径越平滑，但计算量和内存占用也越大。一个 256x256 的 DirectionCell 网格（每个 cell 约 16 字节）需要约 1MB 内存，对于移动端可能偏大。建议根据目标平台和地图大小动态调整分辨率，或使用 分块流场（chunked flow field）仅在玩家视野附近保持高分辨率。

  • Burst + Jobs 的调度开销：并行化并非免费。对于小地图（如 32x32），单一 Job 的调度和同步开销可能超过实际计算收益。可以考虑将网格划分成若干子区域，分帧计算，或者只在目的地改变时重新计算流场，而不是每帧都算。

  • 动态障碍更新：如果场景中有大量移动障碍，每帧重新生成障碍映射会非常昂贵。优化思路：

    • 只在障碍实际改变其占据的网格时才更新映射
    • 使用 脏标记系统，每帧只处理发生变化的障碍
    • 将动态障碍与静态障碍分开，流场基于静态障碍生成全局方向，动态障碍仅用于局部规避

  • 热力图扩散的复杂度：热力图的扩散算法本质上是带权重的 Dijkstra 或 BFS。如果使用 NativeQueue<int> 作为 open list，每次弹出最小 heat 值需要 O(N) 时间，整体复杂度可能达到 O(N²)。优化方案：

    • 桶排序法（Bucket）：因为 heat 通常是整数或离散值，可以用桶队列实现 O(1) 的插入和弹出
    • 并行扩散：使用类似 cellular automata 的多轮传播，每轮所有格子根据邻居更新自己，迭代次数约为网格直径

  • 多目标场景：如果多个单位组同时移动到不同目标，传统的做法是为每个目标独立生成一套流场，这可能非常耗时。可以考虑：

    • 缓存热点目标：将常见目的地的热力图缓存起来，重复使用
    • 分层流场：先在一个低分辨率网格上计算全局流向，再在局部高分辨率网格上微调
    • 增量更新：当新目的地与旧目的地相近时，复用部分热力图数据

  • 局部避障的计算量：流场只提供全局路径，单位间碰撞需要额外的局部算法。RVO 虽然效果好，但每单位需要进行 O(N) 的邻域搜索，在单位数量极大时成本高。优化：使用空间分区（Grid/Hash）来限制搜索范围，或者每 N 帧才执行一次完整避障计算。

最后，如果以上 CPU 优化还不够，可以考虑 Compute Shader 或 Ray Tracing 将热力图扩散和方向计算放到 GPU 上执行，尤其适合超大网格。现代 Unity 中，Compute Shader 与 Burst 的组合能带来数倍性能提升，但也增加了代码复杂度——权衡之乐，正是工程之美。

尾声

好了，这篇博客的核心内容就算写完了。我们来总结一下：

✅ Flow Field 的核心优势：一次计算，全员使用，特别适合 RTS 中成百上千单位同时移动的场景。

✅ 技术栈亮点：

  • 使用 Unity 的 Burst + Jobs System 并行处理，大大提升性能
  • 用 NativeArray 和 NativeParallelMultiHashMap 管理内存
  • 九宫格细分（sub-cell）提高地形检测精度

✅ 实现流程：网格生成 → Cost 场 → 障碍物映射 → Heat Map → Flow Field → 单位移动
